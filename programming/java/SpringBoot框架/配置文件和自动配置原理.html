# 4.2 其它约定配置文件加载顺序

其它约定配置文件加载顺序-官方文档 (opens new window)

springboot启动的时候会扫描以下位置的application.properties或者application.yaml文件(优先级由低到高如下)

  • classpath根目录（编译后，其实就是resource根目录）

  • classpath根config包（resource/config目录）

  • 项目根目录（如果当前项目是继承/耦合关系的maven项目的话，项目根目录 = 父maven项目的根目录）

  • 项目根目录/config

  • 直接子目录/config（水平复制一个项目，更改配置）

    $ java -jar myproject.jar --server.port=9000  #运行jar文件的时候修改配置，此优先级最高
    1
    $ java -jar myproject.jar --spring.config.location=D:\config/ #指定配置文件的位置
    1

同样的由优先级最高的替换，互相补充
备注：对于项目根目录和项目根目录/config的配置在项目打包后就不见了，一般不会使用这两个

# 4.3 Profile文件加载顺序

Profile：配置文件

SpringBoot框架提供了多profile管理，可以使用profile功能来区分不同环境的配置

Profile官方文档 (opens new window)

语法规则：application-{profile}.properties(.yaml/.yml)

常用命名：开发环境配置文件：application-dev.properties生产环境配置文件：application-prod.properties

application.properties中设置spring.profiles.active=dev 则 配置文件application-dev.properties生效

注解@Profile("dev")表示配置文件为dev的时候生效 @Profile("prod")表示配置文件为prod的时候生效

激活指定profile：

  1. 在配置文件中指定spring.profiles.active=dev
  2. 命令行
java -jar **.jar --spring.profiles.active=dev;
1
  • 通过spring.config.location来改变默认的配置文件，但是不会进行互补
java -jar **.jar --spring.config.location=classpath:application-dev.properties
1
  • 通过spring.config.name来改变默认的配置文件，但是不会进行互补
java -jar **.jar --spring.config.name=application-prod
1

# 5、属性注入

两种方式：@Value、@ConfigurationProperties

  • 通过@Value+SPEL直接绑定springboot配置文件中的值
    • @Value只能一个一个的指定
@Value("${user.name}")
public String name;
1
2
  • 通过@ConfigurationProperties(prefix = "user")
    • @ConfigurationProperties可以批量注入配置文件中的属性
    • 此常用于bean属性和yml配置文件的绑定
    • prefix 可以指定配置文件中某一个节点，改节点中的子节点将自动和属性进行绑定
    • 不支持SpEL
    • 支持jsr-303数据校验：@Validated 【需要确保再类路径上有兼容的JSR-303实现】
@Data
@Component
@ConfigurationProperties(prefix = "user")
public class User {
    /**
     * 用户名
     */
    protected String userName;
    /**
     * 密码
     */
    protected String password;
}
